# 打印功能重构计划文档

## 1. 目标

彻底替换当前基于 CSS `@media print` 和 `print-helper.tsx` 的打印方案。新方案旨在生成**高保真**、**内容可复制/搜索**的 PDF 角色卡，同时保证代码的**可维护性**和**可扩展性**，彻底解决旧方案的各种问题。

## 2. 选定方案：`jsPDF` + `html2canvas` 混合方案

- **核心思想**:
  1.  **视觉层**: 使用 `html2canvas` 将渲染后的角色卡 DOM 元素精准地截图为一张高质量的背景图片。
  2.  **文本层**: 遍历 DOM，提取所有文本内容及其精确的坐标和样式。
  3.  **合 成**: 使用 `jsPDF` 创建一个新的 PDF 文档，先将背景图片铺在底层，然后将提取的文本以**不可见**的形式叠加在图片之上完全相同的位置。

- **最终产出**:
  一份看起来与网页上完全一致，但其上的所有文字都可以被正常选中、复制和搜索的 PDF 文档，实现了视觉保真度与内容可用性的统一。

## 3. 主要挑战与难点分析

1.  **文本精确定位 (Pixel-perfect Alignment)**:
    -   **挑战**: 这是整个方案最核心的难点。DOM 中 `getBoundingClientRect()` 提供的坐标、`html2canvas` 的渲染输出、`jsPDF` 的坐标体系三者之间需要进行精确的转换和对齐。任何字体、字号、间距的微小偏差都可能导致文本选择区域与视觉上的文字错位。
    -   **策略**: 建立严格的坐标转换管道，进行大量测试和微调。

2.  **动态内容与布局适应 (Dynamic Content & Layout)**:
    -   **挑战**: 角色卡的很多内容（如物品栏、法术列表）是动态增长的，会导致布局变化。文本提取和定位逻辑必须是动态的，不能依赖写死的坐标。
    -   **策略**: 在生成 PDF 的那一刻，实时遍历当前的 DOM 结构来获取信息，而不是预设布局。

3.  **复杂组件结构下的文本提取 (Text Extraction from Complex Components)**:
    -   **挑战**: 角色卡由深度嵌套的 React 组件构成。需要编写一个健壮的工具函数，能够递归遍历指定的 DOM 节点，准确无误地“捞出”所有需要被索引的文本节点，并排除无关内容。
    -   **策略**: 编写一个通用的 DOM 遍历函数，可以根据 data-attributes 或特定 class 来识别和提取文本。

4.  **自定义字体处理 (Custom Font Handling)**:
    -   **挑战**: 为了视觉统一，`jsPDF` 需要加载并使用与网页上相同的自定义字体（`.ttf` 或 `.woff` 文件）。`jsPDF` 的字体加载和注册机制需要正确配置。
    -   **策略**: 将字体文件转为 Base64 或 `jsPDF` 支持的格式，在生成 PDF 前进行注册。

5.  **性能优化 (Performance Optimization)**:
    -   **挑战**: 对一个复杂的多页角色卡进行截图和文本处理，可能是个计算密集型任务，可能会在短时间内阻塞浏览器主线程，导致页面卡顿。
    -   **策略**: 在 UI 中提供明确的“生成中…”加载提示。对于更极致的优化，可以考虑将计算任务放入 Web Worker 中执行。

## 4. 详细实施步骤

### 阶段一：基础设置与概念验证 (Proof of Concept)

1.  **安装依赖**:
    ```bash
    pnpm install jspdf html2canvas
    ```
2.  **创建服务模块**: 在 `lib/` 目录下创建一个新文件 `pdf-export.service.ts`，用于封装所有 PDF 生成逻辑，避免业务逻辑侵入 UI 组件。
3.  **实现基础截图与PDF生成**: 在 `pdf-export.service.ts` 中，实现一个函数，接收一个 DOM 元素 ID，使用 `html2canvas` 对其截图，然后用 `jsPDF` 将图片放入一个单页 PDF 中并导出。
4.  **UI 集成**: 在 `app/page.tsx` 中添加一个新的“导出为 PDF”按钮，并调用上述服务函数。
5.  **概念验证**: 以一个简单的、内容静态的组件（如 `header-section.tsx`）作为目标，验证可以成功生成一个“只有图”的 PDF。

### 阶段二：文本提取与坐标计算

1.  **开发文本提取器**: 在 `pdf-export.service.ts` 中，编写一个递归函数 `extractTextNodes(element)`。
2.  **识别与提取**: 该函数遍历 `element` 的所有子节点，如果是一个文本节点，则记录其 `textContent`。
3.  **获取坐标与样式**: 对每个文本节点，包裹一层 `<span>`（如果它不是独立的元素），然后使用 `getBoundingClientRect()` 获取其相对于父容器的精确 `x`, `y` 坐标。同时，通过 `window.getComputedStyle()` 获取其 `font-size`, `font-family`, `color` 等信息。
4.  **数据结构**: 将提取的信息存为一个对象数组，例如：`{ text: string; x: number; y: number; fontSize: number; ... }[]`。

### 阶段三：混合方案实现

1.  **集成文本层**: 修改第一阶段的 PDF 生成函数。在添加完背景图后，开始添加文本。
2.  **注册字体**: 在 `jsPDF` 实例中，加载并注册项目中使用的自定义字体。
3.  **渲染不可见文本**: 遍历第二阶段生成的文本信息数组。对每一个文本对象：
    -   使用 `doc.setFont()` 和 `doc.setFontSize()` 设置好对应的字体和大小。
    -   最关键的一步：设置文本渲染模式为“不可见”。通过 `jsPDF` 的底层 API 实现：`doc.internal.write('Tr', '3')`。
    -   使用 `doc.text(text, x, y)` 将文本放置在计算出的精确位置。
4.  **测试对齐**: 生成 PDF，反复检查文本选择区域是否与背景图中的文字完美对齐。这是最需要调试的一步。

### 阶段四：多页与动态内容处理

1.  **分页逻辑**:
    -   将 `character-sheet.tsx`, `character-sheet-page-two.tsx` 等每个“页面”作为独立的截图单元。
    -   遍历每个页面组件，对每个页面分别执行“截图 -> 提取文本 -> 添加到 PDF”的流程。
    -   每处理完一个页面组件，调用 `doc.addPage()` 在 PDF 中创建一个新页面。
2.  **处理可滚动内容**:
    -   对于物品栏等可滚动区域，在截图前，通过临时修改 CSS 的方式，将其容器的 `overflow` 设置为 `visible`，高度设置为 `auto`，让所有内容都展示出来。
    -   使用 `html2canvas` 对这个“展开后”的完整内容进行截图。
    -   截图和文本提取完成后，将样式恢复原状。

### 阶段五：UI 集成与旧方案移除

1.  **添加加载状态**: 在“导出为 PDF”按钮的 `onClick` 事件中，加入 `isLoading` 状态，向用户显示一个加载动画或提示。
2.  **替换旧功能**: 待新方案稳定后，彻底移除旧的“打印”按钮和 `window.print()` 调用。
3.  **代码清理**:
    -   删除 `app/print-helper.tsx` 文件。
    -   在 `app/layout.tsx` 或其他地方移除对 `PrintHelper` 的引用。
4.  **样式清理**:
    -   全局搜索并删除 `globals.css` 及其他组件样式文件中所有的 `@media print { ... }` 代码块。

## 5. 风险与缓解措施

-   **风险**: 文本对齐的调试工作量远超预期。
    -   **缓解**: 严格遵循“概念验证 -> 小范围实现 -> 逐步推广”的路径。保留一个“仅导出图片”的简单模式作为保底方案。

-   **风险**: 性能问题导致在低配置设备上体验不佳。
    -   **缓解**: 优先保证功能的完整实现。性能优化作为后续迭代项，可以引入 Web Worker 或对截图/文本提取算法进行优化。在 UI 上做好充分的等待提示。

## 方案二：【推荐】使用 `html2canvas` + `jsPDF` 实现混合式 PDF 导出

此方案是目前社区中实现高质量、高保真度 Web 页面转 PDF 的最佳实践。

### 方案分析

**可行性分析:**

该方案技术上成熟且可行，能从根本上解决现有问题。

*   **优点:**
    1.  **高保真度 (High Fidelity):** `html2canvas` 通过“截屏”DOM，能像素级还原复杂的 CSS 样式和布局，确保 PDF 视觉效果与浏览器中完全一致。
    2.  **功能完整:** 通过叠加不可见的文本层，最终产出的 PDF 同时具备位图的“所见即所得”和矢量图的“文本可操作性”（选择、搜索、复制），是体验最佳的方案。
    3.  **跨浏览器一致性:** 完全脱离 `window.print()` API，消除了浏览器差异带来的所有不可控因素。

*   **难点与挑战:**
    1.  **文本层精确定位:** 这是技术核心和主要难点。必须编写健壮的逻辑来遍历 DOM，获取每个文本节点的精确位置 (`getBoundingClientRect`) 和样式 (`getComputedStyle`)，然后在 jsPDF 中以 1:1 的精度复现。
    2.  **性能:** 对于包含大量 DOM 节点的复杂组件，`html2canvas` 的渲染过程可能耗时较长。需要提供明确的用户反馈（如加载动画）并考虑性能优化。
    3.  **资源加载:** 自定义字体、图片等外部资源必须在 `html2canvas` 执行前加载完毕，否则无法正确渲染。
    4.  **分页逻辑:** 需要精确计算和分割页面内容，确保分页位置正确，避免内容被截断。

### 详细实施计划

我们将采用分阶段、迭代的方式来实施此方案，以管理复杂性并确保每个阶段都有可交付的成果。

---

#### **Phase 1: 基础单页图像导出 (验证核心流程)**

**目标:** 验证 `html2canvas` 和 `jsPDF` 可以协同工作，生成基础的、仅包含图像的 PDF。

1.  **安装依赖:**
    ```bash
    pnpm install html2canvas jspdf
    ```
2.  **创建导出器组件:** 新建 `components/export/pdf-exporter.tsx`，用于封装所有 PDF 导出逻辑。
3.  **添加触发按钮:** 在 `app/page.tsx` 中添加一个新的“导出PDF”按钮，并将其 `onClick` 事件绑定到导出器组件的导出函数。
4.  **实现基础渲染:**
    *   在导出函数中，获取角色卡主容器的 DOM 元素引用 (`ref`)。
    *   调用 `html2canvas(element)`，传入该 DOM 元素。
    *   在返回的 Promise 中，获取 `canvas` 对象。
    *   将 `canvas` 转换为 `PNG` 格式的 `dataURL` (`canvas.toDataURL('image/png')`)。
    *   创建一个新的 `jsPDF` 实例。
    *   使用 `pdf.addImage()` 将 `PNG` 图像添加到 PDF 中。
    *   调用 `pdf.save('character-sheet.pdf')` 触发下载。

**验收标准:** 点击按钮后，可以下载一个单页的、内容为角色卡截图的、文本不可选择的 PDF 文件。

---

#### **Phase 2: 多页文档生成**

**目标:** 正确处理分页，将多页的角色卡渲染到对应的 PDF 页面。

1.  **识别页面组件:** 确定 `CharacterSheet` 组件中代表不同物理页面的子组件（如 `CharacterSheetPageOne`, `CharacterSheetPageTwo` 等）。
2.  **修改导出逻辑:**
    *   将导出函数修改为异步函数，并显示一个加载指示器（例如 `toast('正在生成PDF...')`）。
    *   创建一个 `jsPDF` 实例。
    *   按顺序遍历每个页面组件的 DOM 元素。
    *   对每个页面 DOM 单独调用 `html2canvas`。
    *   在 `addImage` 之后，如果不是最后一页，则调用 `pdf.addPage()` 来创建新页面。
3.  **调整尺寸:** 确保 `jsPDF` 的页面尺寸（如 A4）与 `html2canvas` 渲染的图像宽高比匹配，避免图像被不当缩放或裁剪。

**验收标准:** 生成的 PDF 包含多个页面，每页对应角色卡的一个页面，布局正确。

---

#### **Phase 3: 实现不可见文本层 (核心功能)**

**目标:** 在图像层之上，添加一个精确对齐的、不可见的文本层。

1.  **创建文本提取函数:**
    *   编写一个递归函数 `extractTextNodes(element, containerRect)`。
    *   此函数遍历指定 DOM 元素 (`element`) 的所有子节点。
    *   **对于文本节点 (node.nodeType === 3):**
        *   获取其文本内容 `node.textContent`。
        *   为了获取其精确位置，需要将其包装在一个 `<span>` 中，然后调用 `getBoundingClientRect()`。计算其相对于容器 (`containerRect`) 的 `x`, `y` 坐标。
        *   获取其父元素的 `window.getComputedStyle()`，记录 `font-size`, `font-family`, `font-weight`, `color`, `line-height` 等。
        *   将 `{ text, x, y, styles }` 对象存入数组。
    *   **对于元素节点 (node.nodeType === 1):** 递归调用此函数。
2.  **修改 PDF 生成逻辑:**
    *   在 `Phase 2` 的基础上，对每个页面，在 `addImage` 之后，执行以下操作：
    *   调用 `extractTextNodes` 获取该页面的所有文本数据。
    *   遍历文本数据数组。
    *   对于每个文本对象：
        *   使用 `pdf.setFont()`、`pdf.setFontSize()` 等方法设置与 `styles` 匹配的字体样式。
        *   **关键步骤:** 设置文本颜色为透明，或使用 jsPDF 的文本渲染模式（如果支持）来实现不可见效果。例如 `pdf.setTextColor(0, 0, 0, 0)` (r,g,b,alpha)。
        *   使用 `pdf.text(text, x, y)` 将文本放置在先前计算出的精确位置。

**验收标准:** 生成的 PDF 视觉上无变化，但现在可以用鼠标选择、复制和搜索其中的文本。

---

#### **Phase 4: 整合、优化与清理**

**目标:** 完成最终集成，提升用户体验，并移除废弃代码。

1.  **用户体验优化:**
    *   添加健壮的加载状态和错误处理。在生成期间禁用导出按钮，结束后隐藏加载提示。如果生成失败，向用户显示错误信息。
2.  **最终测试:**
    *   在主流浏览器（Chrome, Firefox, Safari）上进行全面测试。
    *   使用包含大量数据和不同卡片的角色卡进行压力测试。
    *   打印生成的 PDF，检查物理打印效果。
3.  **代码清理:**
    *   在确认新方案稳定可靠后，可以安全地移除旧的打印方案。
    *   删除 `app/print-helper.tsx` 组件。
    *   从 `app/globals.css` 和其他组件样式中移除所有 `@media print` 相关 CSS 代码块。
    *   将旧的打印按钮替换为新的 PDF 导出按钮。

**风险缓解:** `Phase 3` 的开发调试周期可能较长。可以考虑先上线 `Phase 2` 的版本（高保真但文本不可选），作为一个有效的中间版本，快速解决当前最大的打印不一致问题。然后将“文本选择”作为后续功能迭代。
