### 🏗️ 基础概念

#### 什么是卡牌包？
卡牌包是一个包含多张自定义卡牌的集合，包括：
- **基本信息**：包名、版本、作者等元数据
- **字段定义**：定义你要使用的职业、种族、领域等
- **卡牌数据**：具体的卡牌内容

#### 卡牌包的组成部分
```
卡牌包 = 元数据 + 自定义字段定义 + 各类型卡牌数据
```

一个完整的卡牌包由三个核心部分构成，它们相互依赖，缺一不可：

**1. 元数据 (Metadata)**
- **作用**：定义卡牌包的基本信息和身份标识
- **包含字段**：`name`、`version`、`description`、`author`
这部分帮助用户识别和管理不同的卡牌包，并且提供版本管理、作者信息、包描述等重要信息

**2. 自定义字段定义 (customFieldDefinitions)**  
- **作用**：定义卡牌包的"词汇表"，声明所有将要使用的自定义内容
就像编程中的变量声明，卡包中部分字段必须先定义才能使用。比如职业名称、种族名称、领域名称、扩展卡牌类型等。这是为了确保数据一致性，防止引用未定义的内容。

**3. 卡牌数据 (Card Data)**
- **作用**：具体的卡牌内容，是卡牌包的实际价值所在
- **包含类型**：`profession`、`ancestry`、`community`、`subclass`、`domain`、`variant`
- **依赖关系**：所有卡牌中引用的字段都必须在第二部分中预先定义

### 🌱 最小示例

这是一个只包含一张职业卡牌的最简单例子：

```json
{
  "name": "我的第一个卡牌包",
  "version": "1.0.0",
  "author": "我的名字",
  
  "customFieldDefinitions": {
    "professions": ["元素法师"],
    "domains": ["元素", "魔法"]
  },
  
  "profession": [
    {
      "id": "my-first-pack-1.0-prof-elementalist",
      "名称": "元素法师",
      "简介": "掌控火、水、土、气四大元素的法师",
      "领域1": "元素",
      "领域2": "魔法",
      "起始生命": 12,
      "起始闪避": 8,
      "起始物品": "元素法杖",
      "希望特性": "**元素感知** - 能感受到周围的元素能量",
      "职业特性": "**元素掌控**：你可以操控基础元素。"
    }
  ]
}
```

**重要提示**：
- 在 `customFieldDefinitions` 中声明的名称必须与卡牌中使用的完全一致
- 每张卡牌都需要一个唯一的 `id`

## 第二部分：基础知识

### 📦 卡牌包元数据

每个卡牌包都需要包含基本信息：

```json
{
  "name": "我的奇幻冒险包",           // 卡牌包名称
  "version": "1.0.0",              // 版本号（推荐使用语义化版本）
  "description": "包含魔法和龙的主题卡牌包",  // 简短描述
  "author": "你的名字 <email@example.com>"  // 作者信息
}
```

### 🔧 自定义字段定义 (customFieldDefinitions)

这是卡牌包的"词典"或"字典"，**必须先定义，后使用**。它就像是编程中的变量声明，确保数据的一致性和完整性。

#### 🎯 核心作用

**为什么需要自定义字段定义？**
1. **数据一致性**：确保所有卡牌引用的名称完全一致，避免拼写错误
2. **系统验证**：导入时系统会检查引用的字段是否已定义
3. **类型管理**：特别是扩展卡牌类型，需要定义其结构和属性
4. **用户界面**：系统会根据定义生成筛选器和分类界面

#### 📋 基本结构详解

```json
"customFieldDefinitions": {
  // === 标准字段类型 ===
  "professions": ["职业名1", "职业名2"],     // 自定义职业列表
  "ancestries": ["种族名1", "种族名2"],      // 自定义种族列表  
  "communities": ["社群名1", "社群名2"],     // 自定义社群列表
  "domains": ["领域名1", "领域名2"],         // 自定义领域列表
  
  // === 扩展卡牌类型 ===
  "variantTypes": {                        // 复杂的扩展类型定义
    "类型名称": {
      "description": "类型描述",
      "subclasses": ["子类别1", "子类别2"],
      "levelRange": [1, 10]  // 可选：等级范围
    }
  }
}
```

#### 🔍 字段类型详细说明

##### **标准字段类型**
这些是简单的字符串数组，对应游戏中的基础概念：

**`professions` (职业)**
- **数据类型**：字符串数组
- **作用**：定义卡牌包中所有的职业名称
- **使用场景**：职业卡牌的 `名称` 字段，子职业卡牌的 `主职` 字段
- **示例**：`["星术师", "月影猎手", "元素法师"]`

**`ancestries` (种族)**  
- **数据类型**：字符串数组
- **作用**：定义卡牌包中所有的种族名称
- **使用场景**：血统卡牌的 `种族` 字段
- **示例**：`["星裔", "月之子", "半龙人"]`

**`communities` (社群)**
- **数据类型**：字符串数组
- **作用**：定义卡牌包中所有的社群名称
- **使用场景**：社群卡牌的 `名称` 字段
- **示例**：`["星辰学院", "月影圣殿", "魔法学院"]`

**`domains` (领域)**
- **数据类型**：字符串数组
- **作用**：定义卡牌包中所有的领域名称
- **使用场景**：职业卡牌的 `领域1`、`领域2` 字段，领域法术卡牌的 `领域` 字段
- **示例**：`["星辰", "月影", "预言", "幻象"]`

##### **扩展卡牌类型 (variantTypes)**
这是最复杂也是最灵活的部分，用于定义自定义的卡牌类型：

```json
"variantTypes": {
  "人物": {
    "description": "NPC、盟友、敌人等角色卡牌",
    "subclasses": ["盟友", "敌人", "中立", "商人"]
  },
  "物品": {
    "description": "武器、防具、饰品等装备",
    "subclasses": ["武器", "防具", "饰品", "消耗品"],
    "levelRange": [1, 10]
  },
  "法术": {
    "description": "独特的法术效果",
    "subclasses": ["攻击法术", "防御法术", "辅助法术"],
    "levelRange": [1, 5]
  }
}
```

**每个扩展类型包含的字段**：

**`description` (类型描述)**
- **数据类型**：字符串
- **作用**：内部记录，说明这个类型是做什么的
- **显示**：不会在游戏界面中显示，纯粹用于开发者理解
- **建议**：写清楚这个类型包含什么样的卡牌

**`subclasses` (子类别列表)**
- **数据类型**：字符串数组
- **作用**：定义该类型下的子分类，用于游戏内筛选和组织
- **显示**：会在游戏界面的筛选器中显示
- **必须性**：必须提供，可以是空数组 `[]`
- **使用场景**：扩展卡牌的 `子类别` 字段

**`levelRange` (等级范围)**
- **数据类型**：两个数字的数组 `[最小值, 最大值]`
- **作用**：定义该类型卡牌的等级范围
- **可选性**：可选字段，不是所有类型都需要等级
- **使用场景**：适用于有等级概念的卡牌，如武器、装备等
- **示例**：`[1, 10]` 表示等级1到10

#### ⚡ 使用规则和最佳实践

##### **命名规范**
- **一致性**：确保定义的名称与卡牌中使用的完全一致（包括空格、标点）
- **唯一性**：同一类型内的名称不能重复
- **清晰性**：使用简洁明了的名称，避免过长或难以理解

##### **引用检查**
卡牌数据中的以下字段必须在对应定义中存在：
```json
// 职业卡牌
"领域1": "星辰",  // 必须在 domains 中定义
"领域2": "预言",  // 必须在 domains 中定义

// 血统卡牌  
"种族": "星裔",   // 必须在 ancestries 中定义

// 子职业卡牌
"主职": "星术师", // 必须与某个职业的名称一致

// 领域法术卡牌
"领域": "星辰",   // 必须在 domains 中定义

// 扩展卡牌
"类型": "人物",   // 必须在 variantTypes 中定义
"子类别": "盟友", // 必须在对应 variantTypes.人物.subclasses 中定义
```

##### **常见错误示例**
```json
// ❌ 错误：定义了但拼写不一致
"customFieldDefinitions": {
  "domains": ["星辰", "月影"]
},
"profession": [{
  "领域1": "星晨"  // 错误：应该是"星辰"
}]

// ❌ 错误：引用了未定义的内容
"customFieldDefinitions": {
  "domains": ["星辰"]
},
"profession": [{
  "领域1": "月影"  // 错误：月影未在domains中定义
}]

// ✅ 正确：定义和使用完全一致
"customFieldDefinitions": {
  "domains": ["星辰", "月影"]
},
"profession": [{
  "领域1": "星辰",
  "领域2": "月影"
}]
```

#### 🛠️ 开发工作流建议

1. **先设计概念**：在写卡牌之前，先列出你要用到的所有职业、种族、领域等
2. **完善定义**：在 `customFieldDefinitions` 中完整定义所有概念
3. **创建卡牌**：创建具体卡牌时，只使用已定义的概念
4. **验证一致性**：定期检查是否有拼写错误或未定义的引用

### 🏷️ 卡牌ID命名规范

每张卡牌都需要一个全局唯一的ID。

#### 推荐格式
```
{包名}-{作者}-{版本}-{类型}-{唯一标识}
```

#### 示例
```json
"id": "fantasy-pack-zhang-1.0-prof-mystic-mage"
```

解释：
- `fantasy-pack`：包名简写
- `zhang`：作者名
- `1.0`：版本号
- `prof`：卡牌类型（profession）
- `mystic-mage`：唯一标识

| 卡牌类型 | 中文 |

| profession | 职业 |
| ancestry | 血统 |
| community | 社群 |
| subclass | 子职业 |

## 第三部分：卡牌类型详解

### 📊 卡牌类型概览

DaggerHeart 支持 6 种卡牌类型：

| 类型 | 中文名称 | 用途 | 特殊性 |
|------|----------|------|--------|
| profession | 职业卡牌 | 角色的主要职业 | 定义角色的基础能力 |
| ancestry | 血统卡牌 | 角色的种族背景 | 每个种族有两张卡牌（类别1和2） |
| community | 社群卡牌 | 角色的出身背景 | 提供社会关系和技能 |
| subclass | 子职业卡牌 | 职业的专业化发展 | 有等级要求（基石/专精/大师） |
| domain | 领域法术卡牌 | 特定领域的法术 | 有等级和回想值 |
| variant | 扩展卡牌 | 自定义内容 | 最灵活，可以是任何内容 |

### 🔧 字段详细说明

#### 职业卡牌 (profession)
```json
{
  "id": "必填-字符串",
  "名称": "必填-字符串",
  "简介": "必填-字符串", 
  "领域1": "必填-字符串（必须在domains中定义）",
  "领域2": "必填-字符串（必须在domains中定义）",
  "起始生命": "必填-数字",
  "起始闪避": "必填-数字", 
  "起始物品": "必填-字符串",
  "希望特性": "必填-字符串",
  "职业特性": "必填-字符串",
  "imageUrl": "可选-字符串"
}
```

#### 血统卡牌 (ancestry)
```json
{
  "id": "必填-字符串",
  "名称": "必填-字符串",
```

```json
  "描述": "必填-字符串",
  "imageUrl": "可选-字符串"
```

#### 子职业卡牌 (subclass)
```json
{
  "id": "必填-字符串",
  "名称": "必填-字符串", 
  "描述": "必填-字符串",

  "子职业": "必填-字符串",
```

### ⚠️ 重要约束

#### 施法属性限制
`施法` 字段必须是以下值之一：
- `"力量"`、`"敏捷"`、`"灵巧"`、`"风度"`、`"本能"`、`"知识"`、`"不可施法"`

#### 子职业等级限制
`等级` 字段必须是以下值之一：
- `"基石"`、`"专精"`、`"大师"`

#### 字段一致性要求 
- 职业的 `领域1`、`领域2` 必须在 `customFieldDefinitions.domains` 中定义
- 子职业的 `主职` 必须与某个职业的 `名称` 完全一致
- 领域法术的 `领域` 必须在 `customFieldDefinitions.domains` 中定义
- 扩展卡牌的 `类型` 必须在 `customFieldDefinitions.variantTypes` 中定义

## 第四部分：内容创作指南

### ✍️ 文本格式化

所有卡牌的文本字段都支持 Markdown 格式，系统会自动处理格式化。

#### 基本格式
- `**粗体文本**` - 用于强调重要的游戏术语
- `*斜体文本*` - 用于状态名称或特殊术语
- `\n` - 单行换行
- `\n\n` - 段落分隔

#### 特性标题格式
使用 `*__特性名称__*` 格式标记特性或能力：
```markdown
*__星光指引__*：你的法术在夜晚威力增强。
*__龙血传承__*：你对火焰伤害有抗性。
```

#### 列表项格式
使用 `- ` (连字符+空格) 创建列表：
```markdown
- *__舒缓曲__*：你和近距离范围内的所有盟友**恢复 1 生命点**。
- *__史诗乐__*：使近距离范围内的一个目标暂时陷入 *脆弱* 状态。
```

#### 多特性分段
多个特性之间使用 `\n\n` 分隔：
```markdown
*__激昂演说__*：每长休一次，你可以发表一次发自肺腑、鼓舞人心的演说。

*__诗人之心__*：在你试图打动、说服或冒犯某人时，你可以**花费 1 希望点**。
```

#### 游戏术语强调规范
- 数值和点数：`**生命点**`、`**压力点**`、`**希望点**`
- 状态效果：`*脆弱*`、`*眩晕*`、`*恐惧*`
- 距离范围：`**临近**`、`**近战**`

### 🖼️ 卡牌图片

#### 图片字段基础
每张卡牌都可以设置 `imageUrl` 字段：
```json
{
  "id": "my-pack-prof-mage",
  "名称": "神秘法师",
  "imageUrl": "/my-card-pack/profession/mystic-mage.webp"
}
```

#### 图片规范建议
- **格式**：推荐 `.webp`（体积小，质量高）
- **尺寸**：建议 400×600 像素（卡牌比例）
- **大小**：建议每张不超过 100KB

#### 自动路径推断
如果不设置 `imageUrl`，系统会自动推断：
```
/{卡牌包名称}/{卡牌类型}/{卡牌标识}.webp
```

**推断规则**：
- **卡牌包名称**：内置卡牌用 `builtin-cards`，自定义卡牌用包名
- **卡牌类型**：`profession`、`ancestry`、`community`、`subclass`、`domain`、`variant`
- **卡牌标识**：
  - `ancestry` 类型：使用 `种族` 字段
  - 其他类型：使用 `名称` 字段
  - 转换规则：移除特殊字符，保留中英文数字

#### 图片存储建议
1. **本地存储**：放在 `image` 目录对应文件夹
2. **CDN托管**：使用图床服务，填写完整HTTP链接
3. **文件命名**：使用卡牌中文名称，确保与推断规则一致

### 🔄 内容一致性检查

创建卡牌包时请确保：

#### 字段引用一致性
- [ ] 职业的领域必须在 `domains` 中声明
- [ ] 子职业的主职必须与职业名称一致  
- [ ] 领域法术的领域必须在 `domains` 中声明
- [ ] 扩展卡牌的类型必须在 `variantTypes` 中定义

#### ID唯一性
- [ ] 每张卡牌的ID全局唯一
- [ ] ID符合命名规范
- [ ] 没有与现有卡牌冲突的ID

#### 必填字段完整性
- [ ] 所有必填字段都已填写
- [ ] 数值字段使用正确的数据类型
- [ ] 枚举字段使用正确的预定义值

## 第五部分：AI 创作助手指南

### 🤖 什么是AI创作助手

AI创作助手是一个专门为DaggerHeart卡牌包制作而设计的智能工具，能够将你的创意文档（Word、TXT、Markdown等格式）自动转换为标准的JSON格式卡牌包。

**AI助手的优势：**
- **自动化转换**：无需手动编写复杂的JSON格式
- **格式规范**：严格遵循卡牌包技术规范，避免格式错误
- **批量处理**：能够同时处理大量卡牌内容
- **智能推断**：对于不完整的信息能够合理推断并标记

### 📋 使用前准备

在使用AI助手之前，请准备以下材料：

#### 必需材料
1. **源文档**：包含你要制作的卡牌内容
   - 支持格式：Word(.docx)、纯文本(.txt)、Markdown(.md)等
   - 内容应包含职业、种族、技能、装备等详细描述

2. **AI创作指南**：项目中的 `AI-卡包创作指南.md` 文件
   - 这是AI助手的使用说明书
   - 包含了所有技术规范和约束条件

#### 推荐准备
- **内容大纲**：清晰的卡牌类型和数量规划
- **参考示例**：查看现有卡牌包作为参考
- **验证清单**：准备好检查生成结果的要点

### 🔧 使用步骤

#### 第一步：整理源文档
```
确保你的文档结构清晰：
- 职业描述完整（包含能力、属性、背景等）
- 种族特性明确（每个种族需要两种不同能力）
- 装备属性详细（等级、效果、类型等）
- 法术描述完整（领域、等级、效果等）
```

#### 第二步：与AI助手对话
**标准对话模板：**
```
我有一份关于 [你的主题] 的DaggerHeart游戏内容文档，需要转换为卡牌包。

请参考 AI-卡包创作指南.md 文件 [粘贴或提交文件]，帮我将以下内容转换为标准的JSON格式卡牌包：

[粘贴或提交你的文档内容]

请注意：
- 我需要 “添加特殊的要求”
```

#### 第三步：验证生成结果
收到AI生成的JSON后，请检查：
- [ ] JSON格式语法正确
- [ ] 所有必填字段完整
- [ ] `customFieldDefinitions` 中的定义与卡牌中的引用一致
- [ ] 数值字段使用数字类型（不是字符串）
- [ ] 子职业命名遵循"子职业名+等级"格式
- [ ] 血统卡牌每个种族有且仅有两张

### ⚠️ 重要注意事项

**使用AI时请务必注意：**

#### 必须人工验证
- ✅ AI会严格遵循技术规范，但仍需人工最终检查
- ✅ 复杂的游戏机制可能需要你提供更详细说明
- ⚠️ 始终要验证JSON格式的正确性
- ⚠️ 检查所有字段引用是否正确定义

### 💡 优化建议

**获得更好AI协助效果的技巧：**

#### 分批处理策略

- **逐步扩展**：确认无误后再添加更多内容
- **分类处理**：可以分别处理职业、种族、装备等不同类型

#### 明确指定要求
- **数量规格**：明确告诉AI需要多少张什么类型的卡牌

#### 迭代改进流程
你可以让AI先生成一部分，然后再生成剩余部分，避免部分AI上下文的限制。

### ❓ 常见问题

#### 导入失败
**问题**：卡牌包导入时出现错误
**解决方案**：
1. 检查JSON格式是否正确
2. 确认所有必填字段都已填写
3. 验证字段引用一致性（如领域名称是否已在customFieldDefinitions中定义）
4. 检查ID是否重复
5. 检查json字段拼写是否正确，如（domain，domains的区别）

#### 图片不显示
**问题**：卡牌图片无法正常显示
**解决方案**：
1. 检查图片路径是否正确
2. 确认图片文件是否存在
3. 验证图片格式是否支持
4. 尝试使用系统自动推断路径

#### 字段验证错误
**问题**：某些字段值不被接受
**解决方案**：
1. 检查枚举字段是否使用了正确的预定义值
2. 确认数值字段使用了数字类型而不是字符串
3. 验证字段名称拼写是否正确
